01 · 서비스 설계

"어떤 모델로 운영해야 할지,
혜택 구조를 어떻게 잡아야 할지 모르겠어요"

사업 목적·예산 규모·이용자 특성에 따라 최적 구조가 달라집니다. 서비스 유형, 혜택 방식, 한도 설정의 세 단계로 지역에 맞는 구조를 설계할 수 있습니다.

Step 1 — 서비스 유형 선택
유형 A
복지형 바우처택시
이동약자의 이동권을 보장하는 복지 서비스
이런 경우 적합
  • 특별교통수단 대체·보완이 필요한 경우
  • 장애인·고령자 등 이동약자 대상 복지 사업
  • 이동약자지원센터가 운영 주체인 경우
사례: 경기도 의왕시
유형 B
공공형 택시
대중교통 취약지역 주민의 이동을 보장하는 서비스
이런 경우 적합
  • 농촌·오지 등 버스 노선이 없는 지역
  • 마을 단위 이동 지원이 필요한 경우
  • 가구 단위 바우처 관리가 필요한 경우
사례: 전남 영암군 100원택시
유형 C
관광형 택시
외지 관광객의 이동 편의를 높여 지역 관광을 활성화하는 서비스
이런 경우 적합
  • 관광 인프라는 있지만 이동 수단이 부족한 경우
  • 관광재단·문화재단이 운영 주체인 경우
  • 관광 시즌 집중 운영이 필요한 경우
사례: 경북 영덕군 관광 바우처택시
Step 2 — 혜택 방식 선택
혜택 방식 A
정액 할인 (n원)
호출 1건당 고정 금액 차감
이런 경우 적합
  • 이용자 부담금을 예측 가능하게 유지하고 싶은 경우
  • 지원금 규모를 건당 고정하고 싶은 경우
실제 적용 예시
  • 의왕시: 1회 15,000원 지원, 1일 2회 한도
  • 영덕군: 1회 10,000원 지원, 일 1회·연 3회 한도
혜택 방식 B
정률 할인 (n%)
미터기 요금의 일정 비율 차감
이런 경우 적합
  • 이동 거리 편차가 크고 부담 비율을 일정하게 유지하려는 경우
  • 회당 최대 할인 금액 상한선과 함께 설정 가능
혜택 방식 C
정액 요금
이용자 부담금 고정 (예: 100원)
이런 경우 적합
  • 이용자 부담을 최소화하되 완전 무료는 아닌 경우
  • 정해진 구간만 운행하는 공공형 서비스
사례: 영암군 100원택시
혜택 방식 D
전액 지원
이용자 부담금 0원
이런 경우 적합
  • 이용 장벽을 완전히 없애야 하는 복지 목적
  • 이용자가 결제 부담을 느껴선 안 되는 경우
Step 3 — 한도 설정
한도 유형 A
횟수 한도
  • 1일 최대 이용 횟수
  • 월 최대 이용 횟수
  • 사업 기간 내 누적 최대 횟수
한도 유형 B
금액 한도
  • 회당 최대 지원 금액
  • 월 누적 최대 지원 금액
  • 사업 기간 내 누적 최대 금액
한도 유형 C
기간 한도
  • 발급일로부터 n일 유효
  • 사업 기간 종료 시 자동 만료
  • 특정 날짜 지정 종료
설계 지원 도구
추후 제공 예정
예산 Calculator

예산 규모와 운영 조건(혜택 금액·한도·이용자 수 등)을 입력하면, 실제 운영 데이터를 기반으로 최적의 혜택 구조와 예상 소요 예산을 자동으로 제안합니다.

02 · 기사 모집

"기사님들이 참여할
이유를 어떻게 만들죠?"

셔클 택시는 자율수락형으로 운영됩니다. 기사님이 실제로 느끼는 불편에서 출발해, 셔클이 어떻게 답할 수 있는지 4가지로 정리했습니다.

셔클 기본 배차 방식
조회 기간 내 이용자와
가장 근거리에 있는 택시에게 콜카드 전송
→
가장 빠르게 수락한
기사에게 배차 완료
→
수락 기사 없을 경우
이용자가 여러 번 배차 재시도 가능
기사님의 우려
"정산 요청이 너무 복잡하고, 돈이 들어오기까지 너무 오래 걸려요"
설득 포인트 A
정산사 연계로 선정산·간소화
  • 이즐(정산사)과 연동 — 기사는 운행 완료 후 선정산 수령
  • 지자체는 월 1회 입금만 처리 — 복잡한 건별 정산 불필요
  • 운행 건별 정산 데이터 자동 생성·기록 — 오류 발생 시 즉시 조회 가능
기사님의 우려
"특정 기사에게만 콜이 몰리는 것 같아요. 배차가 공정한지 모르겠어요"
설득 포인트 B
투명한 근거리 배차
  • 이용자와 가장 가까운 택시 순으로 콜카드 전송 — 알고리즘 기반
  • 가장 빠르게 수락한 기사에게 배차 — 누구에게나 동일한 기준 적용
  • 기사별 수락률·콜 이력을 운영툴에서 확인 가능 → 투명성 확보
기사님의 우려
"셔클에만 전념해야 하나요? 기존 영업을 포기해야 하는 건 아닌가요?"
설득 포인트 C
기존 영업 유지 + 추가 매출
  • 타 콜택시 앱과 동시 사용 가능 — 전업 전환 불필요
  • 플로팅 버튼으로 타 앱 이용 중에도 셔클 콜 즉시 확인·수락
  • 배차 효율·사용성 개선 → 이용자 수요 증가 → 기사 매출 증대 → 예산 확대·정책 강화 → 기사 이익 증가 선순환
기사님의 우려
"새로운 앱을 배워야 하고, 입사 절차도 복잡할 것 같아요"
설득 포인트 D
익숙한 방식, 앱 원스톱 심사
  • 카카오T 등 기존 플랫폼과 동일한 콜 수락 구조 — 새로 배울 것이 거의 없음
  • 콜카드에 출발지·도착지·예상거리 한눈에 표시 → 수락 / 거절 한 번으로 처리
  • 기사: 앱에서 심사 지원 → 운영담당자: 관제툴에서 승인 → 즉시 앱 활성화
TBD — 셔클 기사 앱 실사용률 수치 (운영 데이터 확인 후 기재)
03 · 이용자 가입/이관

"디지털에 익숙하지 않은 분들을
어떻게 가입시키죠?"

이용자가 직접 가입하는 것이 어려운 상황에서도 서비스를 시작할 수 있습니다. 누가 등록을 진행하느냐에 따라 적합한 방식이 달라집니다.

이용자 본인
방식 A
앱 직접가입
셔클 앱에서 이용자 본인이 직접 가입
지원 환경
  • Android / iOS 모두 지원
가입 프로세스
  • 카카오톡 · 네이버 소셜 계정으로 연동 가입
  • PASS 통신사 본인인증 (CI 획득)
  • 결제수단 등록
이런 경우 적합
  • 스마트폰 사용에 익숙한 이용자
  • 관광형 등 앱 직접 호출이 기본인 서비스
본인 직접 · 현장 도움
방식 B
간편가입
앱 설치 없이 웹 링크로 가입 — SMS 일괄 발송 또는 현장 태블릿 활용
두 가지 운영 방식
  • 문자 발송: 대상자에게 가입 링크를 SMS·알림톡으로 일괄 발송 → 본인이 직접 진행
  • 현장 태블릿: 행정복지센터·안내소 등에 기기를 비치하고 담당자 도움으로 진행
가입 프로세스
  • 약관 동의 → PASS 본인인증 → 결제수단 등록
  • 인증 완료 시 웹 간편인증 회원으로 등록
이런 경우 적합
  • 앱 설치가 어려운 고령자·장애인
  • 사업 초기 대상자에게 일괄 안내가 필요한 경우
담당자 대행
방식 C
운영주체 대행가입
이용자격 심사 기관(예: 교통약자이동지원센터)이 서류 기반으로 담당자가 대신 등록
등록 방식
  • 담당자가 운영툴에서 원격 회원(RM) 직접 등록
  • 이용자 본인 인증(CI) 없이 처리 가능
  • 연락처(전화번호 또는 유선)만 필요
이런 경우 적합
  • 교통약자이동지원센터 등 이용자격 심사 기관이 존재하는 경우
  • 서류 심사를 이미 완료한 대상자 등록 시
  • 스마트폰 미보유 고령자·취약계층
담당자 대행
방식 D
일괄 등록
사업 초기 대상자 명단 기반으로 계정·바우처를 일괄 생성
전제 조건
  • 이용 약관 및 개인정보 방침에 대한 사전 서면 동의 완료가 필수
이런 경우 적합
  • 사업 개시 시점에 대상자 명단이 이미 확보된 경우
  • 대상자가 많아 개별 등록이 비효율적인 경우
  • 사업 초기 빠른 온보딩이 필요한 경우
04 · 홍보

"바뀐 운영 방식을
이용자·기사님께 어떻게 알리죠?"

새로운 플랫폼 도입 시 홍보 자료 제작이 부담이 됩니다. 셔클은 오프라인 홍보물 디자인을 지원합니다.

지원 항목 A
포스터 · 엑스배너
행정복지센터·관광안내소·역사 등 현장 거점에 비치할 홍보물
셔클 제공
  • 피그마 템플릿 기반 디자인 지원
  • 지역 로고·서비스명 반영 맞춤 제작
  • AI 활용 디자인 제작 가능
지원 항목 B
차량 데칼
셔클 바우처택시 차량임을 식별할 수 있도록 차량 외부에 부착하는 데칼 디자인
셔클 제공
  • 차종별 데칼 디자인 템플릿 보유
  • 지역·서비스 정보 반영 커스텀 제작
  • 데칼 제작 업체 연계 지원
확인 중
디지털 채널 홍보물

SNS 게시물, 앱 내 공지, 문자 발송 등 디지털 채널 지원 방안은 디자인팀 협의 중입니다. 필요하신 경우 담당자에게 별도 문의 부탁드립니다.

05 · 서비스 운영

"매일 처리해야 할 일이 너무 많고
다양한 민원이 발생해요"

오픈 이후 발생하는 운영 업무는 크게 두 영역으로 나뉩니다. 플랫폼 운영툴에서 담당자가 직접 처리하는 영역과, 고객센터(TCK)에 위임해 대행하는 영역입니다.

플랫폼 운영툴 지자체·운수사·운영담당자가 운영툴에서 직접 처리하는 영역
이용자 관리 — 지자체 담당자 / 행정복지센터
이용자 관리
이용자 · 바우처 관리
  • 이용자 검색·조회·상태 확인
  • 바우처 발급·수정·비활성화 및 이력 관리
  • 인증 심사(서류 검토·승인·반려)
바우처 유형
  • 개인바우처: 이용자 1인에게 발급 — 본인만 사용 가능
  • 그룹바우처: 가구 단위 발급 — 가구 내 누구나 공동 사용 가능
    공공형 서비스에서 가구 단위 혜택을 지급하는 경우 활용
이용자 관리
이용자 커뮤니케이션
앱 팝업과 문자(SMS·알림톡)로 이용자에게 일괄 안내할 수 있습니다.
  • 앱 팝업: 서비스 개시·변경사항·혜택 안내를 앱 진입 시 노출
  • 문자 발송: 대상 이용자를 선택해 SMS·알림톡 일괄 발송
  • 담당자가 직접 내용을 작성해 즉시 발송 가능
기사 관리 — 운수사 / 운영담당자
기사 관리
기사 관리
  • 기사별 콜 수락·거절 이력 조회
  • 수락률 낮은 기사 파악 및 운영 정책 적용
  • 기사 온보딩 심사 및 앱 활성화 승인
  • 기사 운행 이력 및 누적 데이터 확인
기사 관리
실시간 관제 현황
관제툴에서 서비스 운행 상황을 실시간으로 확인할 수 있습니다.
  • 참여 중인 차량의 실시간 지도 위치 현황
  • 호출·이동 건수 실시간 집계
  • 배차 대기·운행 중·완료 상태별 현황
  • 이상 상황(배차 지연·미수락 급증) 조기 감지
시스템 관리 — 운영사 / 플랫폼
CS 운영
CS · 민원 역할 분리
  • 이용자 CS: 앱·전화·바우처 관련 → 고객센터
  • 기사 CS: 운행·정산 관련 → 담당 채널 분리
  • 처리 주체를 사전 정의해 현장 혼선 방지
  • 민원 유형별 에스컬레이션 체계 수립 가능
시스템 관리
권한 분리
  • 지자체·운영사·운수사별 접근 권한 차등 설정

06 · 정산

"운행 내역 대조하고 지급 처리하는 데
너무 오래 걸려요"

07 · 예산 집행 관리

"예산을 제대로 쓴 건지 증빙하기 어렵고
언제 소진될지 예측도 안 돼요"

예산 관리는 두 가지 관점에서 다뤄집니다.

증빙 — 적격 대상자에게 올바르게 집행됐음을 사후 입증
예측 — 소진 속도를 일 단위로 파악해 혜택 운영을 선제적으로 조정

관점 1 — 예산 증빙

감사·의회 보고 시 "적격 대상자에게 올바르게 집행됐는가"를 증명해야 합니다. 셔클은 전 과정이 시스템에 자동 기록됩니다.

기록 A
대상자 인증 이력
  • 인증 처리자 · 처리 일시
  • 승인 · 반려 · 만료 결과
  • 근거 서류 첨부 내역
  • 인증 유형 및 유효기간
기록 B
바우처 발급 이력
  • 발급자 · 발급 일시
  • 바우처 현재 상태 (유효·만료·비활성)
  • 수정·변경 이력
기록 C
호출 · 결제 연결
  • 이용자 → 운행 → 결제 → 지원금 흐름 연결
  • 건별 바우처 차감 내역
  • 미수금 발생·해제 이력
기록 D
집계 데이터 내보내기
  • 기간별·이용자별·마을별 집계 조회
  • 감사 대응용 데이터 내보내기
  • 운영툴에서 바로 확인 및 추출 가능
관점 2 — 예산 예측

예산이 얼마나 빠르게 소진되는지 일 단위로 확인하고, 혜택 구조를 유연하게 조정할 수 있습니다.

예측 기능
일 단위 예산 소진 현황
  • 날짜별 바우처 사용 금액 및 건수 추이 확인
  • 현재 소진 속도 기준 잔여 예산 파악 가능
  • 이상 급증·급감 구간 조기 발견
현황에 따라 유연하게 조정 가능
  • 소진이 빠르면 → 혜택 금액·이용 횟수 한도 축소
  • 소진이 느리면 → 혜택 확대 또는 대상자 범위 넓히기
  • 지자체 예산 집행률 목표에 맞춰 실시간 대응 가능
추후 제공 예정
예산 Calculator

예산 규모와 운영 조건(혜택 금액·한도·이용자 수 등)을 입력하면, 실제 운영 데이터를 기반으로 최적의 혜택 구조와 예상 소요 예산을 제안합니다.
